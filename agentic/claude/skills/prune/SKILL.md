# Prune

Identify code, configuration, and files that exist in the repo but are no longer wired up, reachable, or informative. Report every finding with file path and location. Do not delete anything — present findings and wait for confirmation before removing.

The highest-priority category is **AI context bloat**: content that inflates the token cost of reading the repo without improving understanding. A bloated repo means slower, more expensive, and less accurate AI-assisted work. Run this category first, every time.

## Process

1. Identify the tech stack in scope
2. Run **AI Context Bloat** first, then the relevant language/tool checklists
3. For each finding, state: **what** it is, **where** it lives, and **why** it should go
4. Group by category, not by file
5. At the end, present a prioritised removal plan and wait for confirmation before touching anything

---

## AI Context Bloat (run first, always)

These are the highest-leverage removals. Each one reduces the token cost every time an AI reads the file.

### Redundant documentation
- [ ] README sections that repeat information already clear from the code or directory structure
- [ ] Docstrings / comments that restate what the function signature already says (`# increments counter by 1` above `counter += 1`)
- [ ] Multiple docs files covering the same topic from slightly different angles — consolidate or remove the thinner one
- [ ] Changelog entries, migration guides, or release notes embedded in source files rather than git history
- [ ] `docs/` files that describe removed features, deprecated flags, or old workflows

### Bloated comments
- [ ] Commented-out code blocks (dead code hidden in comments is context noise with zero signal)
- [ ] Wall-of-text preamble comments at the top of files that restate the filename or obvious purpose
- [ ] Auto-generated comment headers that add no information (`// Code generated by X. DO NOT EDIT.` followed by 20 lines of boilerplate)
- [ ] TODO/FIXME comments that are months old and clearly abandoned — they occupy context but won't be acted on

### Verbose configuration
- [ ] Config files with every possible key explicitly set to its default value — only non-default values belong in committed config
- [ ] `values.yaml` entries that are placeholder examples rather than real defaults (e.g., `host: example.com`, `password: changeme`)
- [ ] Heavily annotated config files where the annotations duplicate the upstream docs — link to the docs instead

### Large auto-generated or vendored content
- [ ] Auto-generated files that are regenerated on every build and shouldn't be committed
- [ ] Vendored dependency snapshots that belong in a lockfile or registry, not the repo
- [ ] Binary files, rendered PDFs, or pre-built artifacts checked into source
- [ ] `node_modules/`, `.terraform/`, or similar dependency caches committed by mistake

### Duplicate and near-duplicate files
- [ ] Multiple versions of the same template or script (`deploy.sh`, `deploy-v2.sh`, `deploy-new.sh`)
- [ ] Copied config blocks that have drifted — same intent, slightly different content, both present
- [ ] Test fixtures that are identical or near-identical to other fixtures

## Helm

### Unused values
- Enumerate every key path in `values.yaml` (including nested keys)
- Search all files under `templates/` for each key as `{{ .Values.<path> }}`
- Flag any key that has zero template references
- Also check `_helpers.tpl` and any `*.yaml` in subdirectories

### Unused helper partials
- List every `define` block in `_helpers.tpl` (or any `_*.tpl`)
- Search templates for `{{ include "<name>" ... }}` or `{{ template "<name>" ... }}`
- Flag any `define` block that is never included

### Dead template files
- Look for `.yaml` files in `templates/` with no Kubernetes resource kind at the top level
- Flag template files that render to an empty string for all realistic value combinations
- Check for files that are fully commented out

### Leftover scaffolding
- Default `NOTES.txt` content that was never customized
- Placeholder comments like `# TODO: add your resources here`
- Example values in `values.yaml` that are clearly placeholders (`example.com`, `change-me`, `your-name`)

## Terraform

### Unused variables
- List every `variable` block across all `.tf` files in the module
- Search for `var.<name>` references in resources, locals, outputs, and other variables
- Flag variables with zero references

### Unused locals
- List every key in `locals { }` blocks
- Search for `local.<name>` references
- Flag locals with zero references

### Unused outputs
- List every `output` block
- Check if the module is called elsewhere in the repo and whether the output is referenced as `module.<name>.<output>`
- Flag outputs that are never consumed (within the repo — remote consumers can't be verified statically)

### Dead data sources
- List every `data` block
- Search for `data.<type>.<name>` references
- Flag data sources with zero references

### Orphaned variable files
- Check if every `.tfvars` file is actually passed to `terraform apply` (via `-var-file` in Taskfile, CI, or wrapper scripts)
- Flag `.tfvars` files that appear to have no caller

## Go

### Unexported dead code
- Functions, methods, types, and constants that are unexported (lowercase) and have zero references within the package
- Use `grep` patterns: define the symbol, then search for callers in the same package

### Exported dead code (flag only, don't remove)
- Exported symbols with no references anywhere in the repo
- These may be part of a public API — flag for review, not deletion

### Unused struct fields
- Struct fields that are never read or written outside of the struct's own methods
- Fields that are always zero-value and never set

### Dead files
- `.go` files that contain only `package <name>` and no declarations
- Test files with no test functions (`func Test...`)
- Generated files where the generator no longer exists in the repo

---

## Python

### Unused imports
- Imports present but not referenced in the module body
- `from x import y` where `y` is never used
- Wildcard imports (`from x import *`) — flag for explicit replacement

### Unreachable code
- Code after `return`, `raise`, `sys.exit()`, or `continue`/`break` in a loop
- Branches under `if False:` or `if 0:`
- Functions that always raise before returning a value but have a return type hint

### Dead functions and classes
- Functions defined in a module but never imported or called within the repo
- Classes instantiated nowhere in the codebase
- Methods on a class that are never called (excluding dunder methods and those called via `getattr`)

### Leftover scaffolding
- `pass` statements in non-abstract methods that have a body (leftover from stubbing)
- `raise NotImplementedError` in methods that should have been implemented
- Fixture files or test data files in `tests/` not referenced by any test

## General

### Orphaned files
- Config files (`.env.example`, `docker-compose.yml`, `Makefile`) that are never referenced by any script, CI workflow, or documentation
- Files in a `scripts/` or `tools/` directory that are not called from the Taskfile, CI, or README

### Commented-out code
- Blocks of 4+ consecutive commented lines that appear to be disabled code (not documentation)
- Git blame the block — if it has been commented for more than 30 days, it is a prune candidate

### Stale dependencies
- Dependencies declared in `go.mod`, `requirements.txt`, `pyproject.toml`, or `package.json` that are not imported anywhere in the source tree
